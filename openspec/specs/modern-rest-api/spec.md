## Requirements
### Requirement: Versioned resource API

The system SHALL expose the modern REST API under `/api/v1` using plural resource paths, JSON request and response bodies, stable UUIDv7 resource identifiers, RFC 3339 timestamps with offsets, explicit units, and UTF-8.

#### Scenario: Client retrieves a system

- **WHEN** an authorized client requests `/api/v1/systems/{system_id}` with an acceptable media type
- **THEN** the system returns the resource as documented JSON with canonical lowercase UUIDv7 identifiers and explicit timezone/unit semantics

### Requirement: UUIDv7 identifiers

Every identifier generated by PVLog for an entity, event, job, session, audit record, or request correlation SHALL be UUID version 7 and SHALL be represented in modern API payloads and parameters as a canonical lowercase hyphenated UUID string. Clients SHALL treat identifiers as opaque and SHALL NOT infer authorization or authoritative creation time from their sortable bits. External provider subjects, caller-supplied idempotency keys, and imported source identifiers SHALL retain their native formats and map to separate internal UUIDv7 identifiers where persistence requires an internal identity.

#### Scenario: Client submits another UUID version

- **WHEN** a modern API parameter or payload field requiring a PVLog identifier contains a syntactically valid UUID that is not version 7
- **THEN** the system rejects it with a field-specific validation problem and does not perform the requested operation

### Requirement: Consistent problem details

Every modern API failure SHALL use `application/problem+json` compatible with RFC 9457 and include a stable problem type, title, HTTP status, safe detail, request identifier, and field/item errors when applicable.

#### Scenario: Request contains invalid fields

- **WHEN** a JSON request fails validation in multiple fields
- **THEN** one problem response identifies every actionable field violation using stable machine-readable codes

### Requirement: System-derived resource context

System-owned API resources SHALL use the system identifier as their canonical routing context. Clients SHALL NOT repeat an account identifier when the server can resolve and authorize the owning context from the authenticated principal and system registry.

#### Scenario: Client manages an inverter

- **WHEN** an authorized client lists, creates, updates, or deletes an inverter for a system
- **THEN** the operation uses `/api/v1/systems/{systemId}/inverters` and the server resolves the internal ownership context without accepting a redundant account identifier

### Requirement: Bounded collection navigation

Collection endpoints SHALL provide deterministic filtering, sorting, bounded page sizes, opaque cursor pagination, and navigation links or cursors. A cursor SHALL bind to the query shape and stable ordering so concurrent inserts do not silently skip or duplicate already traversed resources.

#### Scenario: Client follows a cursor

- **WHEN** a client requests the next page using an unexpired cursor and the same filters
- **THEN** the system resumes after the prior stable sort key and returns deterministic results within documented consistency semantics

### Requirement: Safe retries and concurrency

Create/bulk endpoints SHALL support `Idempotency-Key` where retries are expected, and mutable resources SHALL expose ETags and honor conditional update/delete headers.

#### Scenario: Stale configuration update

- **WHEN** a client updates a system using an ETag older than the current representation
- **THEN** the system returns a precondition failure and does not overwrite the newer change

### Requirement: Query and bulk ergonomics

The API SHALL support bounded bulk telemetry operations, multi-series time-range queries, explicit aggregation/resolution, selected field inclusion, and asynchronous job resources for work that cannot reliably finish within the request timeout.

#### Scenario: Large export is requested

- **WHEN** an authorized export exceeds the synchronous response threshold
- **THEN** the API returns an inspectable job resource whose eventual artifact has expiry and integrity metadata

### Requirement: Evolution and deprecation

Additive compatible changes MAY occur within `/api/v1`; removals, incompatible type changes, or semantic breaks SHALL require a new major path. Deprecations SHALL be represented in OpenAPI and runtime headers with migration documentation and a published support window.

#### Scenario: Operation is deprecated

- **WHEN** a client calls an operation in its deprecation window
- **THEN** the response includes documented deprecation and sunset metadata while the operation retains its promised behavior

### Requirement: Cache and conditional read semantics

The API SHALL publish explicit cache controls and validators appropriate to resource sensitivity and freshness. Private data SHALL NOT be publicly cacheable.

#### Scenario: Unchanged resource is revalidated

- **WHEN** a client performs a conditional GET with the current ETag
- **THEN** the API returns `304 Not Modified` without a representation body and preserves private cache controls

### Requirement: Stable simple uploader endpoints

The API SHALL accept telemetry uploads only on the canonical system observation routes using `Authorization: Bearer` with an account API key containing `telemetry:write`. It SHALL NOT expose a custom upload-key header, credential-bearing push URL, or query-parameter credential transport. The canonical routes SHALL retain their existing request and response telemetry schemas.

#### Scenario: Bearer endpoint is documented

- **WHEN** an account user or uploader opens the API documentation
- **THEN** the documentation provides a copyable `curl` example using `Authorization: Bearer`, JSON content type, idempotency key, and the canonical system observation URL

#### Scenario: Legacy upload-key transport is attempted

- **WHEN** a client submits `x-pvlog-api-key` or calls a former credential-bearing push route
- **THEN** the API does not authenticate that credential transport

#### Scenario: Scoped bearer client ingests telemetry

- **WHEN** a client uses a valid account API key with `telemetry:write` on the canonical ingestion endpoint for a system owned by its account
- **THEN** the API accepts the request without requiring a separate system ingestion key

### Requirement: Current-user profile resource

The API SHALL expose cookie-authenticated profile read and update operations at `/api/v1/account/profile` without accepting a user or account identifier. Reading SHALL return only the current user's identifier, email, and display name. Updating SHALL accept only a display name and SHALL return the updated safe profile. Failures SHALL use the standard problem-details representation.

#### Scenario: Current user reads their profile

- **WHEN** an authenticated user requests the current-user profile
- **THEN** the API returns that user's safe profile without credential or authorization internals

#### Scenario: Current user updates their profile

- **WHEN** an authenticated user submits a valid display name with CSRF protection
- **THEN** the API returns the updated profile and does not change the login email

### Requirement: Current-account API-key lifecycle resource

The API SHALL expose cookie-authenticated API-key collection operations at `/api/v1/account/api-keys` and revocation at `/api/v1/account/api-keys/{apiKeyId}` without accepting an account identifier from the client. Creation SHALL accept a name, a non-empty unique set of documented scopes, and an optional future expiry; listing and revocation SHALL never return secret material. All failures SHALL use the standard problem-details representation.

#### Scenario: Owner creates a least-privilege key

- **WHEN** an authenticated owner posts a valid name and `telemetry:write` scope
- **THEN** the API returns `201`, safe credential metadata, and the cleartext bearer key exactly once

#### Scenario: Owner lists keys

- **WHEN** an authenticated owner gets the API-key collection
- **THEN** the API returns deterministic safe metadata for only that account and no digest or cleartext key field

#### Scenario: Owner revokes a key

- **WHEN** an authenticated owner deletes one of the account's API-key resources
- **THEN** the API returns `204` and the credential can no longer authenticate

#### Scenario: Request submits an unknown scope

- **WHEN** creation contains an unsupported scope or no scopes
- **THEN** the API returns an actionable validation problem and creates no credential

### Requirement: Operation-specific bearer scope enforcement

Every bearer-enabled operation SHALL document and enforce its required scope using `systems:read`, `systems:write`, `telemetry:read`, or `telemetry:write`. Account ownership and optional system restrictions SHALL be checked in addition to the action scope.

#### Scenario: Telemetry writer attempts system configuration

- **WHEN** a credential containing only `telemetry:write` calls a system update operation
- **THEN** the API returns a generic authorization problem and does not modify the system

#### Scenario: Telemetry reader queries PV data

- **WHEN** a credential containing `telemetry:read` queries telemetry for a system owned by its account
- **THEN** the API permits the query without requiring `systems:write`
