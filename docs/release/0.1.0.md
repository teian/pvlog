# PVLog 0.1.0 release notes

PVLog 0.1.0 is the first stable self-hosted foundation: system aggregates over
inverters and effective-dated strings, canonical telemetry and corrections,
segments/rollups/statistics, SQLite and PostgreSQL storage, local/OIDC/OAuth2
authentication with hierarchical RBAC, alerts/webhooks,
an accessible English/German web application, and operator backup tooling.

## Certified evidence

- Rust formatting, check, Clippy, unit/integration, database-feature, migration,
  fault, recovery, and security suites run with warnings denied.
- Frontend lint/typecheck, component coverage gates, Playwright keyboard/axe,
  production build and bundle budget are enforced.
- OpenAPI lint, 78-operation bidirectional Axum coverage, normalized contract
  comparison, documentation checks, and deterministic capacity reports pass.
- Release inputs are recorded in `release/checksums.txt`; the SPDX document is
  `release/pvlog-0.1.0.spdx.json`.

## Known limitations

- SQLite is an isolated self-hosting profile, not a performance-equivalent
  substitute for the certified PostgreSQL fleet model.
- PostgreSQL backup creation/restoration remains owned by pinned deployment
  `pg_dump`/`pg_restore` hooks; PVLog packages and verifies their archives.
- Regional supply requires an explicitly configured licensed provider and
  degrades to unavailable when none is configured.
- Some optional analytics, notification, and managed-resource
  routers require their production service adapter to be included by the
  deployment composition; their domain/application/API contracts are complete.
- The locally packaged API reference is the supported generated-client source;
  pre-generated language SDK packages are not shipped in 0.1.0.

## Support and deprecation

The latest stable minor receives fixes. Additive API changes are preferred;
removal or semantic changes require a major contract version and a changelog
entry with replacement and removal dates. Security changes may shorten the
window. Operators should attach version, request/job IDs, redacted logs, backend
profile, and relevant health results to support requests, never secrets or raw
personal telemetry.
